### Fix 2 — `ac8681c` partial refunds preserve entitlement; full refunds soft-revoke UserCase

- **Applied:** yes. 3 file changes in one commit.
  - `app/api/webhooks/stripe/route.ts`: replaced the entire
    `handleChargeRefunded` function (lines 380-444 pre-fix). New shape:
    after the existing payment-intent + Order lookup, branch on
    `charge.amount_refunded === charge.amount` (the `isFullRefund`
    boolean). Partial path: a single `prisma.order.updateMany` with
    `where.status: { in: [COMPLETE] }` precondition flips Order to
    `PARTIALLY_REFUNDED`. Idempotent — re-delivery matches zero rows.
    Returns immediately. Full path: a `$transaction` with three
    `updateMany` calls — Order → REFUNDED with `where.status: { in:
    [COMPLETE, PARTIALLY_REFUNDED] }`, ActivationCode revoke with
    `where.revokedAt: null`, UserCase soft-revoke (`revokedAt: new
    Date()`) with the same null precondition. Crucially replaces the
    prior `userCase.deleteMany` with `userCase.updateMany({ data: {
    revokedAt } })` — preserves TheorySubmission + CheckpointAttempt
    history per the operator's product call.
  - `lib/enums.ts`: added `PARTIALLY_REFUNDED: "PARTIALLY_REFUNDED"`
    to the `OrderStatus` const mirror so the new value is callable
    from client components and edge-runtime modules.
  - `tests/api/stripe.test.ts`: extended hoisted mocks with
    `orderCount`, `activationCodeUpdate`, `activationCodeUpdateMany`,
    `userCaseDeleteMany`, `userCaseUpdateMany`. Wired them into the
    `vi.mock("@/lib/prisma")` block and into the
    `transactionFn.mockImplementation` callback. Added a new
    `describe("charge.refunded handler (F-02)")` block with 4 tests:
    full refund (Order REFUNDED, code revoked, UserCase soft-revoked,
    deleteMany NOT called), partial refund (Order PARTIALLY_REFUNDED,
    no transaction, entitlement intact), idempotent full refund
    (zero-row matches everywhere, no errors), partial-then-full

### Fix 4 — `5aace4d` per-recipient activation-email throttle (3/hour)

- **Applied:** yes. 2 file changes in one commit.
  - `app/api/webhooks/stripe/route.ts`: inserted the throttle block
    in `handleCheckoutCompleted` after the activation-code mint
    transaction closes and `if (!updatedOrder.activationCode) return;`
    fires, but before the Resend send. Counts Orders with the
    normalized buyer email + status COMPLETE + emailSentAt within
    the last hour. If `>= 3`, writes `emailLastError: "Throttled: ..."`,
    emits a `[EMAIL-THROTTLE]` warn line, and returns before the
    Resend send — the activation code is already minted and persisted,
    only the email is skipped.
  - `tests/api/stripe.test.ts`: added a default
    `mocks.orderCount.mockResolvedValue(0)` in `beforeEach` so
    untargeted tests are unaffected. New test in the existing
    "POST /api/webhooks/stripe" describe asserts that with 3 prior
    sends, the activation code is still minted, but `resendSend` is
    NOT called and `orderUpdate` writes `emailLastError`.
- **Diff:** 2 files, +110 / 0.
- `tsc --noEmit`: passed.
- `vitest run`: 183 → 184 (+1 new throttle test).
- **Mental trace:** Attacker controls stolen Stripe tokens, mints
  paid checkouts targeting victim@example.com. First three: emails
  fire normally; victim's inbox grows by three. Fourth: Order
  COMPLETE, code minted, throttle counts 3 prior sends, returns
  before Resend. Operator sees the throttled order in the support
  inbox UI (it has `emailLastError` set, no `emailSentAt`). The
  `[EMAIL-THROTTLE]` warn line lets ops grep for the pattern in
  Vercel logs to spot abuse early.
- **Anomalies:** count uses `emailSentAt: { gt: ONE_HOUR_AGO }`
  (only emails actually sent), not `createdAt: { gt: ONE_HOUR_AGO }`
  (every order in the last hour). This is the right choice — a
  throttled order does not have `emailSentAt` set, so it does not
  contribute to future throttle checks (right behavior — a throttled
  email should not gate further legitimate sends).
