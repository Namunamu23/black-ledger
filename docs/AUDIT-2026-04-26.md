## A — Executive Summary

Black Ledger is a well-structured Next.js 16 App Router application with a mature security posture for its stage of development. The P0/P1 wave (Weeks 1–6) addressed the most dangerous issues: privilege escalation, SSRF, CSRF, open-redirect, access-code ownership bypass, and Stripe orphan recovery. Week 9 smoke tests confirmed the database migration path is clean on fresh Postgres.

**No P0 issues remain open.** One P1-class issue was found during this audit (P2-15 corrected — see §E): `ActivationCode.revokedAt` is never checked in the activate route, so an admin-revoked code can still be used to unlock a case. This is the highest-priority open item.

The remaining open work is P2 wave and below: a minor info disclosure (checkout email leak), data-pollution from theory submissions on solved cases, three missing enum mirrors, a feature stub (support reply), and assorted cleanup items.

**Overall assessment**: Ship-ready for a private beta / first real kit sale after the P2-15 fix. The rest of the P2 wave can be addressed in a focused session before public launch.

## E — Security Findings

### P2-15 — `ActivationCode.revokedAt` not checked in activate route (HIGH)

**File**: `app/api/cases/activate/route.ts`  
**Evidence**: Lines 36–79 of activate route. The route fetches the `ActivationCode`, checks `caseFile.isActive` (line 48) and `claimedByUserId` (line 74), but never reads or tests `activation.revokedAt`.

**Schema confirmation**: `ActivationCode.revokedAt DateTime?` is present in `prisma/schema.prisma` line 151. The revoke endpoint at `app/api/admin/cases/[caseId]/codes/[codeId]/route.ts` sets it correctly (`data: { revokedAt: new Date(...) }`).

**Impact**: An admin who revokes a kit code (e.g., to invalidate a fraudulent order) cannot prevent the holder from redeeming it. The revoke action has zero effect until this check is added.

**Note on original P2-15 description**: The previous memory document incorrectly stated that `AccessCode` (the physical QR artifact code) has `revokedAt`. The actual `AccessCode` model has only `retiredAt`. The real `revokedAt` gap is on `ActivationCode` (kit purchase code) in the activate route.

**Fix** — one guard, one line, immediately after the `isActive` check:
```typescript
// After line 53 in activate/route.ts
if (activation.revokedAt) {
  return NextResponse.json(
    { message: "This activation code has been revoked." },
    { status: 410 }
  );
}
```

**Test to add**: A test in `tests/api/admin-codes.test.ts` or a new `tests/api/activate.test.ts` that mocks `activationCode.findUnique` to return a record with `revokedAt: new Date()` and asserts a 410 response with no UserCase creation.

### A5 — Checkout status leaks buyer email without authentication (MEDIUM)

**File**: `app/api/checkout/status/route.ts`  
**Evidence**: Route returns `{ status: order.status, email: order.email }` with no session check. Any caller who has a Stripe `session_id` can retrieve the buyer's email.

**Impact**: Stripe session IDs are long random strings and not easily guessable, but they are exposed in the success page URL (`/checkout/success?session_id=...`), which can leak via Referer headers, browser history, or analytics. This is a minor info disclosure, not an account takeover risk.

**Fix options** (pick one):
- Option A: Strip `email` from the response entirely (the success page only needs to know `status`).
- Option B: If the session is authenticated, return `email`; otherwise omit it.

Option A is simpler and sufficient for the current UI.

### S11 — CSP `img-src` missing R2 public origin (LOW)

**File**: `next.config.ts`, line 28  
**Evidence**: `img-src 'self' data: blob:` — R2 public URL (`R2_PUBLIC_URL` env var) is not included.

**Impact**: Currently report-only, so no images are blocked today. When the CSP transitions to enforced mode, all hero images (`CaseFile.heroImageUrl`) and person portraits (`CasePerson.portraitUrl`) stored in R2 will be blocked.

**Fix**: Add the R2 origin to `img-src`. Since the R2 URL is an env var, either:
- Hardcode the known production R2 origin in the constant array, or
- Generate the header dynamically in the `headers()` function using `process.env.R2_PUBLIC_URL`.

Also note: There is no `report-to` URI configured, so CSP violation reports go nowhere. Consider adding a report endpoint or using a third-party CSP reporting service before enforcing.

---

### S12 — Image upload validates MIME type by header only (LOW, admin-only)

**File**: `app/api/admin/uploads/sign/route.ts`  
**Evidence**: The route accepts a `contentType` field in the request body (validated against an allowlist of image MIME types), but this value is provided by the client and is not verified against the actual file bytes.

**Impact**: An attacker with admin access could upload a disguised file to R2 (e.g., an HTML file with a `Content-Type: image/jpeg` header). Because this is an admin-only route, risk is low, but the file would be stored on R2 with the spoofed content type.

**Fix (deferred is acceptable)**: Add magic-byte verification on the presigned PUT callback side, or validate during the blurhash endpoint (which does call `sharp` and would fail on non-image inputs anyway).

## F — Correctness Findings

### A1 — TheorySubmission written when UserCase.status is SOLVED (MEDIUM)

**File**: `app/api/cases/[slug]/theory/route.ts`  
**Evidence**: The theory test (`tests/api/theory.test.ts`, lines 103–133) shows that when `status === "SOLVED"`, the route still:
- Calls `theorySubmission.create` (line 126: `expect(mocks.theorySubmissionCreate).toHaveBeenCalledOnce()`)
- Calls `userCase.update` to write back `status: "SOLVED"` and preserve `completedAt`

The existing test verifies that SOLVED is terminal (status doesn't downgrade). It does NOT verify that the submission is skipped entirely.

**Impact**: Every theory submission after a case is solved creates another `TheorySubmission` row. These accumulate silently and pollute the submission history shown on the bureau workspace page.

**Fix**: Add an early return before the evaluation logic:
```typescript
if (ownedCase.status === "SOLVED") {
  return NextResponse.json(
    { message: "Case already solved.", resultLabel: "CORRECT", score: 3 },
    { status: 200 }
  );
}
```

**Test to add**: Assert that when `userCaseFindFirst` returns `{ status: "SOLVED" }`, `theorySubmissionCreate` is NOT called and the response is 200.

### A3 — `resolveContent` missing `hidden_evidence` branch (LOW-MEDIUM)

**File**: `app/api/access-codes/redeem/route.ts` (lines 9–27) and `app/bureau/cases/[slug]/page.tsx` (lines 28–67, `resolveEvidence`)  

**Evidence**: Both functions handle `record`, `person`, and `hint`, then fall through to a raw/null return for anything else. `AccessCode.unlocksTarget` is a JSON blob that can contain `{ type: "hidden_evidence", id: N }`, but `HiddenEvidence` rows are never fetched. Codes with this target silently return `{ type: "hidden_evidence", raw: { type: "hidden_evidence", id: N } }`.

**Impact**: `HiddenEvidence` model and `HiddenEvidenceKind` enum are fully authored in the schema but inaccessible via the QR flow. Any access code targeting a `HiddenEvidence` row effectively does nothing useful.

**Fix**: Add a `hidden_evidence` branch to `resolveContent` and `resolveEvidence` that fetches `prisma.hiddenEvidence.findUnique({ where: { id: target.id } })` and returns `{ type: "hidden_evidence", evidence: { id, title, kind, body } }`. Update `UnlockedPanel` in `UnlockForm.tsx` and `RevealedEvidence` to render the new shape.

---

### A6 — No duplicate purchase guard in checkout route (LOW-MEDIUM)

**File**: `app/api/checkout/route.ts`  
**Evidence**: The checkout route creates a Stripe session and a pending `Order` row unconditionally. There is no check for an existing `UserCase` (case already owned) or a pending/complete `Order` for the same `(caseId, email)` pair.

**Impact**: A user can click "Buy" multiple times or return to the page after payment and buy the same case again. Each click creates a new Stripe session. If multiple sessions are completed, the webhook creates multiple `ActivationCode` rows and the user owns multiple copies.

**Fix**: Before creating the Stripe session, check:
1. If the user is logged in: check for existing `UserCase` with this case.
2. Always: check for an existing `Order` with `status: PENDING | COMPLETE` for this `(caseFileId, email)` pair.

## H — Configuration Findings

### CSP is report-only (known — S11 covered above)

### HSTS preload is aggressive
**File**: `next.config.ts`, line 19  
`Strict-Transport-Security: max-age=63072000; includeSubDomains; preload`

This is a 2-year HSTS with `preload`. Submitting `theblackledger.app` to the HSTS preload list is irreversible across major browsers. Only proceed with preload once you are certain all subdomains will serve HTTPS permanently. Consider removing `preload` for now and adding it deliberately at launch.

### No CSP report-to endpoint
There is no `report-to` or `report-uri` directive in the CSP. Violations accumulate in the browser console but are never collected centrally. Add a `report-to` group header if you want to catch violations in production.

### Prisma `output` path
`prisma/schema.prisma` line 3: `output = "../generated/prisma"`. The generated client lives outside `node_modules`. This is intentional for Prisma 7 and works correctly with the adapter pattern.

## J — Architecture Findings

### ARC-1 — Legacy aggregate PUT coexists with per-section PATCH endpoints (INFO)

**Files**: `app/api/admin/cases/[caseId]/route.ts` (PUT, ~490 lines) and the 6 PATCH endpoints (`overview`, `people`, `records`, `hints`, `checkpoints`, `solution`)

The legacy `PUT /api/admin/cases/[caseId]` still exists and is functional. It and the per-section PATCHs both write to the same models and both emit `CaseAudit` rows. The tab components in the admin editor now use the PATCH endpoints exclusively. The PUT is no longer called by any UI but remains reachable via API.

**Recommendation**: Keep for now (it's harmless and provides a fallback). Mark it as deprecated in a comment and remove in a cleanup pass once the PATCH endpoints are battle-tested.

### ARC-2 — `CaseAudit` not written for several admin actions (INFO)

The following admin operations do not create `CaseAudit` rows:
- Workflow transitions (`PATCH /api/admin/cases/[caseId]/workflow`)
- Batch activation code generation (`POST /api/admin/cases/[caseId]/activation-codes`)
- Activation code revoke (`PATCH /api/admin/cases/[caseId]/codes/[codeId]`)
- AccessCode create (`POST /api/admin/cases/[caseId]/access-codes`)

For a product with kit sales and physical artifact QR codes, audit logging revoke and AccessCode create actions has real operational value (e.g., "which admin revoked code X, and when?"). This is a medium-priority improvement before first real sales.

### ARC-3 — No `retiredAt` PATCH endpoint for AccessCodes (INFO)

`ActivationCode` has a revoke endpoint. `AccessCode` (physical QR) has neither a retire endpoint nor admin UI that surfaces the `retiredAt` field. To retire a QR code, an admin would need to do it via the database directly. Noted in CLAUDE.md P2.

### ARC-4 — `resolveEvidence` duplicated between redeem API and bureau page (INFO)

The logic in `resolveContent` (`access-codes/redeem/route.ts`) and `resolveEvidence` (`bureau/cases/[slug]/page.tsx`) are near-identical. Consider extracting to `lib/resolve-evidence.ts` when adding the `hidden_evidence` branch (A3) — doing it twice in two places would be the third instance.

## K — Performance Findings

### BureauCasePage — N sequential DB lookups for redemptions

**File**: `app/bureau/cases/[slug]/page.tsx`, lines 138–142  
```typescript
const revealedEvidence = (
  await Promise.all(
    redemptions.map((r) => resolveEvidence(r.accessCode.unlocksTarget))
  )
).filter(...)
```

`Promise.all` fires all DB lookups concurrently, which is correct. The lookups are `findUnique` by PK on `CaseRecord`, `CasePerson`, or `CaseHint`. For typical case sizes (10–30 evidence items), this is fine. For very large cases, batch-fetching (one `findMany` per type, grouped by ID) would reduce round trips.

### BureauPage — Separate `count()` for theory submissions

**File**: `app/bureau/page.tsx`, lines 38–42  
A separate `prisma.theorySubmission.count({ where: { userId } })` is issued. This is a minor concern — it could be consolidated with the `userCase.findMany` query if needed. Acceptable for now.
